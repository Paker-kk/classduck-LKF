# 任务50：课程删除联动彻底物理删除修复

## 1. 任务背景
1. 用户复测反馈：课程主页面删除手动课程后，待办主页面仍存在关联待办残留。
2. 需求明确：删除课程时，必须联动删除该课程关联的全部待办（未完成/已完成都删），且为物理删除，不允许仅标记状态。
3. 本次修复要求：只修删除联动，不回归改动无关功能；修复后需重新模拟器联调。

## 2. 本次改动

### 2.1 待办仓储删除逻辑升级为“规范化匹配+物理删除”
- 文件：`classduck_app/lib/features/todo/data/todo_repository.dart`
1. 新增课程名规范化函数 `_normalizeLinkedCourseName`：
   - 去首尾空格
   - 统一小写
   - 去除全角/半角空白
   - 去除历史尾部括号补充信息（如 `课程名(周一1-2节)`）
2. `deleteTodosByCourseName` 改为返回删除数量 `Future<int>`。
3. 原生端改为先扫描 `todo_item` 的 `id/course_name`，按规范化课程名精确筛选后逐条物理删除。
4. Web 端内存列表也改为同一规范化匹配规则。

### 2.2 课程改名联动一致性增强
- 文件：`classduck_app/lib/features/todo/data/todo_repository.dart`
1. `renameCourseName(from, to)` 改为按规范化课程名匹配历史记录后更新，避免脏数据导致改名联动漏改。

### 2.3 课程删除链路回显增强
- 文件：`classduck_app/lib/features/schedule/ui/schedule_page.dart`
1. 删除课程时接收待办联动删除数量：
   - `final int removedTodos = await _todoRepository.deleteTodosByCourseName(course.name);`
2. 删除成功提示改为包含联动删除条数，便于现场验收：
   - `已删除课程：xxx，联动删除待办 N 条`

## 3. 执行规则映射（对应用户要求）
1. 触发条件：课程详情二级界面点击删除并确认后，立即执行联动删除。
2. 执行范围：扫描并删除全量待办记录，不区分未完成/已完成。
3. 操作要求：对匹配记录执行数据库物理删除（非标记完成）。
4. 校验标准：课程删除后，待办页不应再出现该课程关联记录；删除提示可见联动删除数量。

## 4. 验证计划
1. 在模拟器执行链路：
   - 新建课程 A
   - 新建 2 条关联课程 A 的待办（分别置于未完成/已完成）
   - 删除课程 A
   - 核验待办两分栏均无残留
2. 保留删除提示中的联动数量，作为快速自检线索。

## 5. 风险与后续
1. 由于当前待办与课程仍按“课程名字符串”关联，若未来引入课程唯一 ID 关联，可进一步消除同名课程的潜在歧义。
2. 本次修复已尽量兼容历史脏数据（空格/括号后缀），用于收敛现阶段残留问题。
